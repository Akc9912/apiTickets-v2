# TODO Triage Backend Skill

## Goal

Turn raw TODO comments into a prioritized, actionable backlog with clear ownership and ticket readiness for Java/Spring Boot backend projects. Can also generate sprint-ready task lists from implementation plans.

## Inputs

**Mode A - TODO Comments:**

- One or more files, controllers, services, repositories, or entities.
- Standardized TODO comment format for tree-sitter/tooling compatibility:

```
// P0 SECURITY [service] Fix JWT validation bypass in AuthService
// P1 REFACTOR [repository] Replace N+1 query with JOIN FETCH
// P2 TECHDEBT [controller] Add request validation annotations
// P3 DOCS [api] Update OpenAPI schema annotations
```

**Format:** `// P0-P3 TYPE [area] Description...`

- **Priority:** P0 (critical), P1 (important), P2 (debt), P3 (cleanup)
- **Type:** SECURITY, BUG, PERF, REFACTOR, FEATURE, TEST, DOCS
- **Area:** [controller], [service], [repository], [entity], [dto], [config], [security], [api], [mapper], [db], [validation]

**Mode B - Implementation Plan:**

- Path to `docs/plan/plan-XXX.md` generated by ImplementPlan skill
- Creates actionable sprint tasks per phase

## Workflow

**Mode A - TODO Comments:**

1. Scan target files for TODO comments.
2. Normalize comments to one-line actionable format.
3. Remove duplicates and merge equivalent tasks.
4. Re-rank priorities using impact (P0-P3).
5. Mark ticket candidates (P0/P1, cross-component, >4h).

**Mode B - Sprint Tasks from Plan:**

1. Read `docs/plan/plan-XXX.md` from ImplementPlan.
2. For a specific phase (or all phases sequentially):
   - Break down high-level tasks into actionable sprint tasks.
   - Assign priority (P0-P3) based on phase criticality.
   - Tag with area (`[service]`, `[controller]`, `[repository]`, `[security]`, etc.).
   - Estimate effort (hours).
3. Generate one sprint document per phase with:
   - Sprint goal for that phase
   - Tasks for that phase only (prioritized P0-P3)
   - Definition of done for that phase
   - Dependencies on previous sprints

## Area Tags (Backend-Specific)

- `[controller]` - REST controllers, endpoints, request handling
- `[service]` - business logic services, domain services
- `[repository]` - data access layer, JPA repositories
- `[entity]` - JPA entities, domain models
- `[dto]` - data transfer objects, request/response DTOs
- `[config]` - configuration classes, beans, properties
- `[security]` - authentication, authorization, JWT, Spring Security
- `[api]` - external API clients, integrations
- `[mapper]` - entity/DTO mappers (MapStruct, manual)
- `[db]` - database migrations, schema changes, indexes
- `[validation]` - input validation, constraints, validators

## Type Tags

- `SECURITY` - Vulnerabilities, auth bypasses, data leaks, injection issues
- `BUG` - Functional defects, race conditions, null pointer exceptions
- `PERF` - Query optimization, caching, lazy loading issues
- `REFACTOR` - Code structure improvements, SOLID violations, tech debt
- `FEATURE` - New functionality additions
- `TEST` - Unit tests, integration tests, test coverage
- `DOCS` - Javadoc, README, API documentation

## Output Format

**Mode A - TODO Report:**

- P0 list
- P1 list
- P2/P3 list
- Ticket candidates (title + area + rationale)

**Mode B - Sprint Document:**

Generates `docs/sprint/sprint-XXX.md` (incremental numbering) - **one sprint per phase**:

````markdown
# Sprint N: [Phase Name]

**Based on:** docs/plan/plan-XXX.md - Phase N  
**Date:** YYYY-MM-DD  
**Duration:** X hours  
**Dependencies:** Previous sprint (if any)

## Sprint Goal

[Phase objective]

## Tasks

### Critical (P0)

```java
// P0 [area] Actionable task with acceptance criteria (Xh)
```
````

### Important (P1)

```java
// P1 [area] Actionable task with acceptance criteria (Xh)
```

### Optional (P2/P3)

```java
// P2 [area] Optional task (Xh)
```

## Definition of Done

- [ ] Criterion 1
- [ ] Criterion 2

## Next Sprint

- **Blocks:** sprint-YYY.md (Phase N+1)

```

## Guardrails

- Keep comments short and concrete.
- Do not add implementation details inside TODO lines.
- Avoid adding TODO comments in files that are purely stable infrastructure (e.g., common utils, base entities).
- Prefer service-level TODOs over method-level for maintainability.
- Follow AGENTS.md TODO format: `// P0 TYPE [area] description`
- Documentation only: do not implement code changes from this skill output.
```
